# Stackdose.App.DeviceFramework 方案說明

## 1. 專案定位

`Stackdose.App.DeviceFramework` 是設備型 WPF 應用的「組裝框架層」。
它的主要價值是用 JSON 設定驅動 App 啟動、Shell 導航、設備頁資料綁定與 PLC 命令流程，讓新設備專案可以低程式碼上線。

一句話：**這不是單一設備 App，而是用來快速打造多設備 App 的框架。**

## 2. 技術與相依

- Framework: `net8.0-windows`
- UI: `WPF`
- 主要相依專案：
  - `..\Stackdose.UI.Core\Stackdose.UI.Core.csproj`
  - `..\Stackdose.UI.Templates\Stackdose.UI.Templates.csproj`
  - `..\Stackdose.App.ShellShared\Stackdose.App.ShellShared.csproj`

## 3. 目錄地圖（快速理解）

- `Services/`: 框架核心（啟動、導航、Runtime、映射、命令）
- `Models/`: Config 與 Runtime 資料結構（`AppMeta`, `MachineConfig`, `DeviceContext`）
- `Pages/`: 通用設備頁（`DynamicDevicePage`）
- `ViewModels/`: 通用頁面 ViewModel 與命令封裝
- `docs/`: 使用指南（目前檔案有編碼異常，建議以程式碼實作為準）

## 4. 核心執行流程

1. App 端在 `MainWindow` 建立 `AppController`。
2. `AppController.Start()` 呼叫 `BootstrapService.Start(...)`。
3. `RuntimeHost.Start(...)` 自動尋找 `Config` 目錄，載入 `app-meta.json` 與 `Machine*.config.json`。
4. `RuntimeMapper` / `DeviceContextMapper` 將 JSON 轉為 runtime context（含 Labels、Commands、Modules）。
5. `ShellNavigationService` 註冊標準頁面導航（Overview/Detail/Log/User/Settings）。
6. `NavigationOrchestrator` 根據操作切換頁面；Detail 頁可用 `DynamicDevicePage` 直接承接。
7. 若啟用 `EnableMetaHotReload`，`MetaRuntimeService` 監聽 `app-meta.json` 並即時套用導航/標題變更。

## 5. 關鍵類別職責

- `Services/AppController.cs`: App 生命週期主控（Start/Stop/Dispose），外部專案主要接觸點。
- `Services/RuntimeHost.cs`: 載入 Config、建立 `RuntimeContext`、初始化 Overview。
- `Services/BootstrapService.cs`: 啟動時注入 Shell 命令與導覽處理器。
- `Services/NavigationOrchestrator.cs`: 統一頁面切換（Overview/Detail/Log/User/Settings）。
- `Services/DeviceContextMapper.cs`: `MachineConfig` -> `DeviceContext`，並將 `tags.status/process` 的 read 項目自動注入 Labels。
- `Services/ProcessCommandService.cs`: 寫入 PLC 命令地址並回傳執行結果。
- `Pages/DynamicDevicePage.xaml(.cs)`: 以 `DevicePageViewModel` 動態呈現 Labels/Commands，不需 app 自訂頁面也可運作。

## 6. Config 驅動規則（最重要）

- `app-meta.json`：控制 Header、預設標題、導航項目、Hot Reload、Overview 顯示策略。
- `Machine*.config.json`：每台設備一份；定義 `machine/plc/commands/processMonitor/detailLabels/modules/tags`。
- `detailLabels` 與 `tags` 都可形成頁面 Label，且 `detailLabels` 同名 key 優先。
- `commands` 會自動變成可點擊按鈕，並由 `ProcessCommandService` 執行。

## 7. 擴充點（App 客製化入口）

優先順序建議：

1. **先用純 JSON + `DynamicDevicePage`**（最快上線）。
2. 需要特殊映射規則時，覆寫 `IRuntimeMappingAdapter` / `DefaultRuntimeMappingAdapter`。
3. 需要特化資料上下文時，覆寫 `RuntimeMapper.CreateDeviceContext(...)`。
4. 需要客製 UI 時，用 `AppController.ConfigurePageFactory(...)` 換成自訂 Device Page。
5. 需要客製 Settings 時，設定 `AppController.SettingsPage` 與 `OnSettingsNavigating`。

## 8. 專案方向與建議

- 方向：持續把「設備差異」收斂到 JSON 與 Adapter，保持框架通用性。
- 優先實務：
  - 新專案先走最小啟動路徑，避免一開始就重寫頁面。
  - 變更導航與標題優先改 `app-meta.json`，不要硬寫死在 code-behind。
  - PLC 地址與流程事件命名統一透過 Config 與 `ProcessEventNames` 管理。

## 9. 已知風險/注意事項

- `docs/DeviceFramework-Guide.md` 目前有亂碼（疑似編碼不一致），教學內容需以 `Services/*`、`Models/*`、`Pages/*` 實作為準。
- `DynamicDevicePage.xaml` 仍有硬編碼色碼；若要與 Core 設計規範一致，建議後續改為主題 Token。
- 若 `Config` 路徑與輸出目錄配置不一致，`RuntimeHost` 可能載入到非預期配置；發版時需固定配置來源。

## 10. 30 秒摘要（給交接/AI）

`Stackdose.App.DeviceFramework` 是設備 App 的框架層：用 `AppController` 管生命週期，用 `RuntimeHost` 載入 JSON，用 Mapper 把設定變成 `DeviceContext`，再由 `DynamicDevicePage` 動態顯示資料與命令。你可以先用純設定快速啟動，再按需求覆寫 Adapter/Mapper/頁面，做到「先上線、後精修」的可演進架構。
